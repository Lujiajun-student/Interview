# 1. char和varchar的区别

char定长，存储时剩余的空位会被占用。varchar是变长，第一个字节存储数据长度，随后才存储数据。

# 2. decimal和float与double的区别

float和double是浮点数，只能存储近似的小数值。但decimal是定点数，采用十进制字符串的方式存储，存储对精度有要求的小数值，避免浮点数带来的精度损失。

# 3. datetime和timestamp的区别

datetime没有时区信息，timestamp有时区。

timestamp只需要使用4个字节来存储信息，但datetime需要使用8个字节来存储。但datetime能够存储0000年到9999年的事件，而timestamp在32位系统中只能存储1970到2038年，超出后会出现问题，但64位系统就没问题。而且timestamp可以进行自动的时区转换。

# 4. NULL和''的区别

NULL是缺失或者不确定的结果，而''表示一个存在的空字符串。

NULL不能等于自己，像`NULL = NULL`的结果是NULL不是true或者false，任何值与NULL比较返回的结果都是NULL，表示结果的不确定。而''可以和其他字符串一样比较，结果会返回true或者false。

# 6. MySQL索引

MySQL索引是一种用于快速查询的数据结构。索引结构有B树、B+树、红黑树等。但MySQL使用的是B+树。

索引的优点：

1. 提高查询速度。通过索引可以减少扫描的数据量，减少磁盘IO次数。
2. 保证数据的唯一性。创建唯一索引可以保证表中的某一列唯一，如用户ID等，主键本身就是一种唯一索引。
3. 如果查询有ORDER BY等排序字段，索引本身已经排序，所以可以避免额外的排序操作。

索引的缺点：

1. 创建和维护耗时。除了维护数据库，还要额外维护索引表。而对数据进行增删改时，也要维护索引表，降低DML的执行效率。
2. 占用额外空间。索引也是一种数据结构，以物理文件的格式存储，占用一定的空间。
3. 可能会失效。如果查询语句写不好，数据库优化器可能不会使用索引，导致性能下降。

B+树的优势：

1. B+树的特点是高度低。索引树的高度可能只有3-4层，也就是只需3-4次IO就可以查找到数据，而全表扫描可能需要逐个数据来检索。
2. B+树的叶子节点用链表连起来，找到开头后，就能顺着链表顺序一直获取数据。

## 6.1 索引分类

1. 主键索引。唯一索引，不允许空值。一张表只有一个主键。
2. 唯一索引。列中的值必须唯一，但允许为空。
3. 普通索引。基本的索引，可以包含单列，也可以包含多列。（Where name 是单值索引，Where name and age需要联合索引）
4. 全文索引。用于在大量文本中寻找关键字，而不是等值匹配。

还有的按物理存储来分类。

1. 聚簇索引。索引即数据，叶子节点包含完整的行记录。
2. 二级索引。叶子结点存储的是主键的值。如果通过二级索引查找数据，会先在二级索引树中找到主键，再去聚簇索引树中招完整的行数据。这就是回表查询。

# 7. 为什么InnoDB不使用哈希作为索引结构

哈希作为索引结构有很多缺点。

1. 不支持范围查询。哈希函数会把相邻的输入值映射到不相邻的位置，这样就无法实现范围查询，哈希只能退化为全盘扫描。
2. 不支持排序。同样的，无法使用哈希索引来优化ORDER BY子句。
3. 不支持部分索引查询。如联合索引，哈希必须使用联合索引的所有子段进行查询，无法单独利用左边的索引加速查询。
4. 存在哈希冲突。如果出现哈希冲突，需要额外的链表来存储数据，降低性能。

# 8. 为什么不使用B树

B树的每个节点都会存储全部数据，而B+树只有叶子结点存储数据，对于同一个大小的页面，B+树可以容纳更多的节点。也就是说，同样的数据量，B+树会更矮，查找数据需要的IO次数更少。

此外，B树必须中序遍历才能保证顺序，如果查询多个节点，需要在树的不同层级之间反复回溯，IO压力大。而B+树先通过根节点定位到范围起始的叶子结点，然后顺着链表顺序向后遍历即可获取整个范围的所有顺序，不需要从根节点再次出发。

B+树查询性能稳定。B+树的每一次查询都一定会走到叶子结点，查询路径长度固定，但B树有可能根节点就找到数据，有可能到叶子结点才能找到，性能不稳定。

# 9. 什么是覆盖索引

如果一个索引包含所有需要查询的字段的值，那么就是覆盖索引。

一般来说，非主键索引的叶子结点包含的是索引值与主键值。如果使用非主键索引来查询，就会先找到主键值和索引值，然后根据主键索引来检索其余需要的数据。这样就查了两次表，称为回表查询。

覆盖索引就是一次查询只需要这个索引树包含的字段，直接根据这个索引就能查找到数据，不需要再根据主键来回表，就是覆盖索引。

# 10. 什么是联合索引

联合索引就是用多个字段来构建索引。

```mysql
ALTER TABLE users ADD INDEX idx_name_age (last_name, first_name, age);
```

上面就是用first_name, last_name和age构建了联合索引。

如果查询这三个字段，可以根据先后顺序，先匹配first_name，然后匹配last_name，最后匹配age来查找数据。

## 10.1 最左匹配原则

使用联合索引时，MySQL会根据索引的字段顺序，从左到右匹配查询条件的字段。如果查询条件与索引的字段从左到右逐个匹配，就会使用索引来过滤数据，提高查询效率。

最左匹配原则会按联合索引字段顺序先后匹配，直到遇到范围查询为止。

```mysql
SELECT * FROM users WHERE last_name = 'Anne' and first_name = 'Henry' and age = 18;
```

这样就会按照索引顺序逐个匹配。

```mysql
SELECT * FROM users WHERE last_name = 'Anne' and age = 18;
SELECT * FROM users WHERE age = 18;
SELECT * FROM users WHERE first_name = 'Henry' and age = 18;
```

上面的语句的查询条件字段顺序与索引顺序不符，不会使用这个联合索引。

```mysql
SELECT * FROM users WHERE last_name = 'Anne';
SELECT * FROM users WHERE last_name = 'Anne' and first_name = 'Henry';
```

上面的语句没有打破索引的顺序，依然可以使用联合索引来检索。

# 11. SELECT * 会使索引失效吗？

SELECT * 不会直接导致索引失效，但由于联合索引很少会包含所有的字段，所以会非常大地提高放弃使用索引的概率，并且会查询不必要的数据，导致不必要的开销。

# 12. 什么字段适合创建索引？

1. 被作为条件查询的字段。比如某个字段经常用来过滤数据，就应该用来作为索引。
2. 频繁查询的字段。如身份证号、手机号等，而且这些字段的值几乎不会重复，用来作为索引效率很高。
3. 需要排序的字段。索引会自动排序，所以查询可以利用索引的排序来加快查询时间。
4. 用于表连接的字段。多表查询时，外键必须建立索引。执行JOIN时，数据库需要从表中查找与主表匹配的行。如果没有索引，每一次匹配都要全表扫描。

# 13. 索引失效的原因

1. 在索引列上进行了计算。索引是基于字段的原始值构建的B+树，如果对字段进行了计算或函数操作，就无法使用索引。

```mysql
SELECT * FROM users WHERE YEAR(birthday) = 2026;
```

2. 隐式类型转换。查询的索引类型与字段定义的类型不一致，会进行类型转换，类型转换也属于函数操作，不会使用索引。

3. 不遵循联合索引最左匹配原则。如果联合索引时不先匹配最左索引，或者改变了索引的顺序，就不会使用索引。

4. 模糊查询的左模糊，也违反了最左前缀原则。

5. OR连接了非索引。这样导致了数据库进行全盘扫描。

```mysql
SELECT * FROM users WHERE id = 1 or name = 'Anne';
```

如果name不属于索引，那么就需要进行全盘扫描。

6. 范围查询之后的字段无法使用索引。

# 14. MySQL查询缓存

在以前，MySQL执行查询语句后，会将结果保存到缓存中。接下来执行查询时，会先查找缓存，如果有结果就直接返回。

但查询缓存有问题，匹配条件非常严格，比如`FROM`和`from`属于不同的关键字，改大小写后不会匹配，而且`SELECT * FROM`和`SELECT  * FROM`也不匹配，只是因为多了一个空格。

查询不命中除了没有事先缓存，还有三种情况。

1. 查询在任何字符上的不同都会导致缓存不命中。
2. 如果查询中包含自定义函数、用户变量、随机值等，无法保证两次查询结果一致，查询结果也不会保存。
3. 缓存建立后，如果表的数据或者结构发生改变，关于这张表的所有缓存会全部丢失。

因此，对于写密集的数据库，缓存会大幅降低性能，因为每次写操作都需要删除大量缓存。因此，查询缓存在8.0之后不支持。

# 15. MySQL常见日志

MySQL主要有三个日志。

1. Binlog。属于Server层，主要记录逻辑日志，也就是SQL语句或者数据行变更情况。作用是主从复制、数据恢复，在事务提交时写入。
2. Redo Log。属于InnoDB层，记录的是物理日志，如页的修改，主要用于故障恢复。
3. Undo Log。属于InnoDB层，记录逻辑日志，如修改的反向操作，用于事务回滚、版本并发控制等。在事务期间修改数据前写入。

# 16. MySQL事务

## 16.1 什么是事务

事务是逻辑上的一组操作，要么都执行，要么都不执行。

数据库事务就是用来保证多个对数据库的操作组成一个整体，要么全部执行，要么全部不执行。

```mysql
START TRANSACTION; # 开启事务

SQL1, SQL2 ...

COMMIT # 提交事务
```

## 16.2 事务的ACID特性

1. 原子性。事务是最小的执行单位，原子性保证事务要么全部完成，要么全部不完成。主要依靠Undo Log来实现回滚。
2. 一致性。执行事务前后，数据需要保证一致，也就是预设的约束以及业务逻辑的一致性。比如转账前后，两人的总额不变。这是ACID的最终目的。
3. 隔离性。并发访问数据库时，事务之间不会互相干扰。
4. 持久性。事务被提交后，对数据库的修改是持久的。

## 16.3 并发事务的问题

1. 脏读。发生在一个事务读取了另一个事务修改但未提交的数据时。如果另一个事务发生了回滚，那么这个事务获取到的数据就是不存在的数据。
2. 不可重复读。指一个事务内多次读取同一条记录，结果却不一致。
3. 幻读。同一事务内多次执行相同的范围查询，前后查询到的数据行数不一致。

## 16.4 并发事务的解决方式

SQL定义了四种隔离级别。

1. 读未提交。级别最低，不能防止脏读、不可重复读、幻读。
2. 读已提交。只能读取已经提交的数据。可以防止脏读，不能防止不可重复读和幻读。
3. 可重复读。保证事务内对同一数据的读取是一致的。能够防止脏读和不可重复读，不能防止幻读。**这是默认的隔离级别**。
4. 串行化。一次只能执行一个事务，能够防止这三个问题，但性能较低。

## 16.5 并发事务的控制方式

1. 锁。控制方式通过锁来控制共享资源。MySQL主要通过读写锁来实现并发控制。

* 共享锁（S锁）。读锁，读取记录的时候共享，允许多个事务同时获取。
* 排他锁（X锁）。写锁，修改记录时获取写锁，不允许其他事务获取排他锁和共享锁，也叫锁不兼容。

2. MVCC多版本控制。MVCC为每一行记录保存多个版本。一个事务读取数据时，读取的是数据的快照，而不是最新版本。使用的手段是隐藏字段、读视图和回滚日志。

* 隐藏字段：每行数据会多两个字段，分别记录创建该版本的事务ID和指向旧版本的指针。
* 读视图ReadView。决定当前事务能够看到哪个版本的数据。
* 回滚日志Undo Log。记录历史版本，创建旧版本的数据。

隔离级别中，串行化是通过锁实现的，读已提交和可重复读的级别是通过MVCC实现的。

## 16.6 MYSQL的表级锁和行级锁

表级锁是粒度比较大的锁，对当前操作的整张表加锁。主要针对非索引字段的锁，实现简单，但冲突率高。

行级锁是粒度最小的锁，对当前操作的行数据加锁。针对索引字段加的锁，能够大幅减少冲突，但加锁开销大，且会出现死锁。

## 16.7 行级锁使用的注意事项

如果WHERE条件不满足索引，就无法根据索引树锁住对应的B+树叶子节点，需要进行全盘扫描。而全盘扫描时，会对扫描到的所有行记录都加锁。

# 18. 如何进行MySQL性能优化

首先应该考虑慢查询。可以先开启慢查询日志，在先配置好`slow_query_log`为`ON`，`long_query_time`设置阈值，可以为1秒，`log_queries_not_using_indexes`记录所有没有走索引的SQL。

还有可以在SQL前使用EXPLAIN来查看优化器如何执行语句。如果type出现ALL，就是全盘扫描，需要优化。而key如果是NULL，说明没走索引。rows是预估扫描的行数，越小越好。额外信息中如果有Using filesort，说明需要进行额外的排序，需要优化，或者Using temporary说明使用临时表，需要优化。

如果没走索引，那么就可以查询条件来创建索引或者使用覆盖索引来避免回表，以及优化SQL语句，避免使用`SELECT *`，使用具体字段来查找。

# 19. 为什么MySQL的表需要使用InnoDB来存储

InnoDB最主要是有ACID特性，支持事务。而MyISAM不支持事务，无法保证多个同时操作的安全性。

InnoDB提供行级锁，能够只锁一行数据，但MyISAM只有表级锁，容易冲突，性能低。

InnoDB有redo log，在宕机后能够进行数据恢复。

# 20. 数据库三大范式

1. 数据库的每一列都是不可分割的原子数据项。
2. 非主键属性必须完全依赖主键。比如选课表存在学生id和课程id为主键，如果还有学生姓名，学生姓名只依赖于学生id，就导致一个学生选两门课，就需要在选课表中重复存储学生姓名两次。需要拆为学生表和选课表。
3. 任何非主键属性不依赖于其他非主键属性。比如学生表中有班主任姓名和班主任年龄，这两个非主键存在依赖关系，需要拆为学生表和班主任表，在班主任表中将班主任姓名和班主任年龄与班主任id联系起来。

# 21. 连表查询

连表查询主要有四种。

1. 内连接INNER JOIN。只返回两个表中完全匹配的行。

```mysql
SELECT s.name, c.class_name
FROM students s
INNER JOIN classes c ON s.class_id = c.id;
```

只有s.class_id和c.id值相同的数据行才会列出来。

2. 左外连接LEFT JOIN。返回左表的所有行以及右表匹配的行。

```mysql
SELECT s.name, c.class_name
FROM student s
LEFT JOIN classes c ON s.class_id = c.id;
```

这里会列出所有学生数据，如果s.class_id 与c.id不匹配，只会显示NULL，否则会显示c.id对应的c.class_name。
