# 1. 为什么Redis快

Redis快的原因有四点。

1. 纯内存操作。Redis数据读写都在内存中，访问速度比数据库快得多。

2. 使用单线程事件循环和IO多路复用，让一个线程能够处理多个网络网络连接的io事件，避免了多线程的上下文切换和锁竞争的问题。

3. 优化的数据结构。Redis提供多种数据类型，采用了优化后的编码方式来储存数据。

4. 高效的通信协议。Redis使用的是RESP协议，序列化和反序列化的开销小。

# 2. 本地缓存与Redis的区别

最主要的原因是数据一致性。多服务器部署的时候会存在数据不一致的问题。而Redis所有的应用实例共享同一个数据，就不会存在分布式环境数据不一致的问题。

然后是内存的限制，本地缓存被进程的堆内存限制，过大的缓存会阻碍垃圾回收导致内存溢出，而Redis可以部署在服务器上，可以使用更多的内存空间。

还有数据持久化，本地缓存如果宕机，数据就会丢失。但Redis可持久化，宕机后数据依然可以恢复。

# 3. Redis的应用

除了做缓存，Redis还有其他的作用。

1. 分布式锁。Java的锁只能锁住当前进程。如果要实现多进程的锁，就要使用SETNX或者Redisson框架。
2. 消息队列。Redis的List数据结构可以作为简单的队列来使用。而且Redis5.0新增的Stream数据结构更适合，可以实现主题订阅或者消费组，也可以实现消息的持久化和确认机制。
3. 限流。为了保护数据库不暴露在大量的流量下，可以使用Redisson的RRateLimiter来实现分布式限流。

# 4. Redis数据类型

Redis有5中基础数据类型和3种特殊数据类型。

基础数据类型是String、List、Set、Hash、Zset。

特殊数据类型是HyperLoglog基数统计、Bitmap位图和Geospatial地理位置。

位图能够用来操作String，节省内存。HyperLogLog用来统计不重复元素的数量。占用的内存只有12KB。地理位置用来存储经纬度，并计算两点之间的距离。

# 5. String和Hash存储对象的区别

String存储对象时，通常会将对象序列化为JSON或者XML等存入Redis。

```
SET user: 1 '{"name":"Alice", "age": 25, "city":"Beijing"}'
```

客户端获取String对象只需要反序列即可。而且可以使用压缩算法来减少内存。但修改较难，如果要修改对象的一个字段，需要将对象反序列，修改后再重新序列化改回去，容易出现数据覆盖的情况。

而Hash存储对象时，将对象的每个属性存储到Hash的field中。

```
HMSET user: 1 name "Alice" age 25 city "Beijing"
```

这样，可以直接修改对象的某一个字段，并发更安全。而且Redis内部对Hash使用ziplist来存储，节省了空间。并且如果只需要对象的某个属性，可以直接取出，不需要读取完整的对象。但如果对象嵌套很深，对象里还有对象，那么Hash处理会比较麻烦。

# 6. 有序集合的底层实现

如果有序集合ZSet保存的数据较少，就会使用压缩列表来存储元素。压缩列表是连续的内存区域，元素按分数从小到大来排列。如果数据量变大，就会使用跳表SkipList来存储。跳表能够实现O(logN)的按序插入、删除和范围查询。

对平衡树和红黑树来说，虽然能够做范围查询，但需要进行中序遍历的操作，实现比较复杂而跳表使用双向链表，找到范围的起点后顺着链表遍历就行。而且为了维护树的平衡，平衡树和红黑树在插入和删除节点是需要进行旋转和变色操作，代码复杂。但跳表用的是随机层数算法来保证代码简洁。

对B+树来说，它的作用是作为数据库或者文件系统的索引结构，通过尽可能少的IO来定位到尽可能多的索引来获取索引数据。但Redis不会存储大量数据，不需要使用B+树来维护。

# 7. Set的应用场景是什么

Redis中，Set是无序集合。主要用于存放数据不能重复的场景，比如网站UV统计、文章点赞等。或者需要获取多个数据源的交集、并集和差集等，比如共同好友、共同粉丝、推荐好友等。

# 8. Redis持久化

## 8.1 RDB持久化

Redis可以创建快照来将存储在内存中的数据在某个时间点上的副本存储起来。Redis默认采用了持久化，在redis.conf中有配置。

```conf
save 3600 1 300 100 60 10000
```

这里分别有三个键值对，分别的意思是3600秒内至少有1个key变化、300秒内有100个key发生变化、60秒内有10000个key发生变化，就会执行持久化。Redis提供了两个命令来生成RDB快照文件。

* save。同步保存，会阻塞Redis线程。
* bgsave。fork新进程，在子进程执行保存操作。

虽然bgsave不会阻塞Redis线程，但fork操作本身就会阻塞，因此需要减少bgsave的频率。

在bgsave过程中如果发生数据修改，就会触发COW机制，把需要写的数据复制出来进行修改，子进程完成了数据持久化之后，再将修改的内容应用到数据文件，这样保证了子进程读取文件的数据一致性。

## 8.2 AOF持久化

AOF主要分为5步。

1. 命令追加。所有的写命令都会追加到AOF缓存。
2. 文件写入。将AOF缓冲区的数据写入到AOF文件中。
3. 文件同步。根据redis.conf配置的appendfsync，Redis会调用fsync来将AOF缓存中的写命令同步到磁盘中。
4. 文件重写。AOF内保存的是命令，占用空间较大，需要定期对AOF文件进行重写。Redis会创建新的AOF文件来读取当前数据库的键值对，生成指令来代表键现在的状态来代替历史指令。
5. 重启加载。如果Redis进行了重启或宕机，可以加载AOF文件来进行数据恢复。

而这里文件同步时，持久化有三种方式。

1. appendfsync always。主线程执行写操作时，会立刻使用fsync来同步AOF文件。这样主线程会阻塞，直到fsync将数据同步到硬盘才会返回。这种不会出现数据丢失的情况，但性能较差。
2. appendfsync everysec。主线程执行写操作后立刻返回，而后台线程aof_fsync会每秒钟盗用一次fsync函数来同步AOF文件。这样性能比较好，而且宕机时最多只丢失1秒的数据。
3. appendfsync no。主线程执行写操作后立刻返回，让操作系统决定什么时候进行同步。这种方式避免了fsync的频繁阻塞，但数据容易大量丢失。

在Redis4.0之后，默认开启了RDB和AOF混合持久化。

# 9. Redis多路复用

虽然Redis是单线程，但通过IO多路复用，能够同时监听客户端的大量连接。Redis会将客户端的Socket连接注册到多路复用器上，如果某个Socket收到了数据包，操作系统就会返回这个数据，让Redis处理有数据的连接。这样，Redis就不需要创建额外的线程来监听大量的连接，降低了资源的消耗。

# 10. Redis6.0之前的单线程

Redis6.0之后，开始使用了多线程。

而Redis4.0时就已经加入了多线程的支持。但Redis4.0增加的多线程是针对删除操作。执行删除命令时，Redis只会将键从哈希表中逻辑删除，然后创建后台线程来释放内存。这样，主线程就能立即返回，处理其他请求。

而Redis使用单线程的原因主要有三个。

1. 单线程比较容易维护，且编程容易。
2. Redis的性能瓶颈不在CPU，而是内存和网络，因此对多线程的需求不大。
3. 多线程会出现死锁、上下文切换等问题，会影响性能。

# 11. Redis6.0之后的多线程

到了Redis6.0后，网络IO的读写变成了性能瓶颈。因此Redis引入了多线程，将网络读写分配给多个IO线程来并发处理，减轻主线程的压力。但Redis本身执行命令依然是在主线程中完成的，不会存在线程安全的问题。

# 12. Redis后台线程

Redis虽然是单线程，但还有一些后台线程用来执行耗时的操作。

* bio_close_file进程用来释放AOF/RDB过程产生的临时文件。
* bio_aof_fsync线程调用fsync将缓冲区未同步的数据同步到磁盘。
* bio_lazy_free线程用来释放已删除对象的内存空间。

# 13. 过期key删除策略

Redis中，过期数据的删除策略有4种。

1. 惰性删除。在取出或者查询key时检查是否过期。对CPU友好，但会导致大量不访问的key无法删除。
2. 定期删除。周期性随机抽查设置了过期时间的key，逐个检查key是否过期。对内存更友好，但CPU有一点负担。
3. 延迟队列。将设置过期时间的key放到延迟队列中，到期就删除。能保证过期key被及时删除，但维护队列需要资源。
4. 定时删除。为每一个设置了过期时间的key设置定时器，时间到达时立即删除。对CPU影响最大。

而Redis默认使用定期删除+惰性删除。定期删除对内存友好，惰性删除对CPU友好，这样能够两者兼顾。

# 14. 内存淘汰策略

Redis如果运行内存达到配置的最大内存阈值时，就会触发内存淘汰策略。

1. volatile-lru。从设置过期时间的数据中挑选最近最少使用的数据来淘汰。
2. volatile-ttl。从设置过期时间数据中选即将过期的数据淘汰。
3. volatile-random。从设置过期时间的数据中随机挑选数据淘汰。
4. allkeys-lru。从所有数据中选最近最少使用的数据淘汰。
5. allkeys-random。从所有数据中随机挑选数据淘汰。
6. no-eviction。禁止淘汰数据，内存满后写入操作会报错。

Redis4.0之后为volatile和allkeys添加了lfu方法，挑选最不经常使用的数据淘汰。

# 15. 事务

Redis内可以开启事务，能够将多个命令进行打包，然后按顺序执行打包的所有命令。

使用MULTI命令后可以输入多个命令，直到调用EXEC后才会执行这些命令。躲着调用DISCARD命令取消MULTI命令后的事务队列中保存的所有命令。

在MULTI开启事务后，可以使用WATCH命令来监视某个Key，如果事务执行时Key被外部线程修改，整个事务就不会执行。

# 16. Redis事务的缺点

关系型数据库的事务有四大特点：原子性、隔离性、持久性和一致性。而Redis事务中如果出现了运行时错误，比如找不到key，或者对key使用了其他类型的操作，Redis就会跳过这个错误命令，执行剩下的命令。因此这种情况下**不满足原子性**。

而Redis的持久化不是针对事务，而是针对保存数据的。事务执行后，数据写入到内存中，而内存数据什么时候写入磁盘，取决于Redis的持久化配置。

Redis的持久化配置有RDB和AOF两种，如果事务执行后立刻宕机，RDB没有时间创建快照，数据会全部丢失。而AOF的持久性取决于它的配置。如果采用always，即每次写命令同步到AOF文件，那么这能够满足持久化，但严重降低了性能。如果采用everysec，每秒执行一次持久化，也会丢失1秒的数据，不满足事务的**持久化**。

为了解决这个事务的缺陷，Redis引入了Lua脚本，批量执行多条Redis命令，这些命令会交给Redis服务器一次性执行完成。由于Redis是单线程，执行脚本时其他命令不会执行，提高了原子性。但Lua脚本如果运行出错，依然无法回滚，仍然不满足原子性。

或者可以采用乐观锁，先WATCH某个key，如果事务执行EXEC前这个key被修改，那么整个事务就不会执行。

# 17. Redis网络传输优化

Redis的命令可以简化为四步。

1. 发送命令。
2. 命令排队。
3. 命令执行。
4. 返回结果。

而发送命令和返回结果需要进行网络传输，耗费的时间为RTT。使用批量操作能够减少网络传输次数来提高效率。

支持批量操作的命令有MGET, MSET, HMGET, HMSET, SADD等。但这种命令在集群操作下无法保证所有获取的Key落在同一个哈希槽中，返回CROSSSLOT错误。

哈希槽：Redis采用无中心化的分片架构，整个集群划分为16384个哈希槽，每个主节点负责其中一部分，无权操作其他主节点的哈希槽。

对不支持批量操作的命令，可以使用pipeline来将Redis命令封装起来，一次性提交到Redis服务器上使用。同样的，会存在CROSSSLOT错误的问题。

# 18. Lua脚本

Redis可以使用Lua脚本来批量操作多条命令。一个Lua脚本可以被当成一条命令来执行，此时其他脚本或命令不会执行。但Lua脚本如果出错，出错命令后的写操作不会撤销，无法回滚。

# 19. BigKey

如果一个key的value占用内存比较大，那么这个key会视作BigKey。如果String类型的value占用达1MB，或者复合类型的value包含元素超过5000个，就会看作BigKey。

BigKey多半是程序设计不当，或者对数据规模考虑不周到。这会导致客户端阻塞事件增加，导致客户端超时，而且返回数据时，会占用大量的网络流量，也容易导致网络阻塞。

通过`redis-cli -p 6379 --bigkeys`能够查看bigkey。

# 20. Hotkey

如果一个key的访问次数较大，这个key可以看作是hotkey。出现的原因是某个热点数据访问量暴增，例如热搜事件、秒杀商品等。

处理Hotkey会占用大量的CPU，影响Redis对其他请求的处理通过`redis-cli -p 6379 --hotkeys`可以查找hotkey。

 如果Hotkey请求超出了Redis的处理能力或者Hotkey过期，那么会有大量的请求进入数据库，导致数据库崩溃。

# 21. 慢查询命令

排除网络传输的情况外，Redis的命令执行步骤也可能耗时很长。Redis大部分命令的时间复杂度为O(1)，但也会存在许多时间复杂度为O(n)的命令，比如返回所有符合条件的key，返回Hash的所有键值对，返回Set所有元素等。这些命令可能会导致命令执行时间过长。因此，可以使用Redis内置的Slow Log来记录执行时间超过阈值的命令。在配置文件redis.conf中，可以使用`slowlog-log-slower-than`来设置阈值，以及`slowlog-max-len`来设置记录慢日志的数量。一般来说，`slowlog-log-slower-than`会设置为10秒钟，`slow-max-len`设置为128。

通过`SLOWLOG GET`就能获取检测到的所有慢命令。

# 22. 内存碎片

Redis调用第三方分配器来管理内存，采用固定空间分配的方法来分配固定的连续空间。比如某个数据只需要5字节，但分配器会按照2的倍数来分配8字节的连续空间。那么这些空余下来的空间不会被其他数据使用，导致空间浪费。

内存碎片的产生主要有两个原因。

1. Redis存储数据的时候向操作系统申请的内存空间会大于数据实际需要的空间。

Redis进行内存分配的时候，不仅要分配数据大小的内存，还会多分配PREFIX_SIZE大小的内存，用来管理这个数据的实际大小。

2. 频繁修改Redis的数据也会产生内存碎片。

数据的变动会引发内存再分配，而内存分配器的回收会存在滞后性。如果对一个key进行修改，而且修改后的大小超过当前的内存块大小，那么Redis必须要申请更大的空间来存储数据。旧的内存块被标记为空闲，但不会立刻返回给操作系统，而是留在Redis的进程中等待复用。而如果接下来的新数据都很大，那么这些就的内存块就无法被利用，导致内存碎片。

通过`info memory`可以查看内存信息。而Redis在4.0版本引入了activedefrag参数，能够在配置文件中开启，并设置清理条件，碎片率或者碎片大小超过一定条件时就会自动清理。

# 23. 缓存穿透

如果大量请求的key不存在缓存中，也不存在数据库中，那么会导致这些请求直接放到了数据库中，不会经过缓存，造成数据库的巨大压力。

为了避免这些情况，首先需要做好参数校验，不合法的参数请求需要直接抛出异常信息给客户端。

1. 缓存无效key。如果缓存和数据库都查不到这个key，就需要到Redis中设置这个key数据为空。这样重复的大量请求就不会落到数据库上。但如果无效的key存在大量类型，会导致Redsi缓存大量无用的key。
2. 布隆过滤器。这个过滤器设置在缓存之前，针对数据库中存在的每一个key都会通过多个哈希函数计算，得到多个索引位置，并将位数组中的对应位置为1。那么接下来查询这个key时，会对这个key进行同样的哈希运算，只有每一个哈希运算后的位数组对应位置都为1，才能判断当前key存在，才会访问Redis和数据库。只要有一个为0，就判断当前key不存在数据库和Redis中，直接返回。
3. 接口限流。对每个用户或者IP进行限流，能够避免数据库被单一用户大量访问。而这个限流也能够解决击穿和雪崩的问题。

# 24. 缓存击穿
