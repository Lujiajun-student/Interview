# 1. Kafka的使用场景有哪些，作用是什么

Kafka也就是消息队列，给出了一个管道，生产者向管道输入数据，消费者向管道获取数据。这样能够实现分布式的解耦，生产者不需要等待消费者准备好才发送数据，消费者不需要查看生产者是否工作才接收数据，生产者和消费者就能直接操作管道来生产和消费数据。最重要的一点就是异步。

这样，异步操作可以有很多使用场景，比如解耦，生产者不需要关注有没有消费者或者哪个消费者消费数据，只需要向队列放入消息即可。还有高峰期的处理，在大量请求来到的时候，只需要将这些请求按顺序放到队列中，后端按自己的节奏来消费，不会出现吞吐量过大的情况。

# 2. Kafka的分区

如果一个Topic只有一个分区，那么数据只能存在一个Broker上，使用不了多个Broker。如果进行分区，那么不同的区可以放到不同的Broker上，不仅减轻了Broker的负担，而且还能实现并行消费数据。这样的话，生产者能够根据给定的分区号来决定存入哪个分区。而消费者消费消息时，同一个分区只能被一个消费组里的一个消费者消费，而不同组的消费者能够同时消费消息。

# 3. 基本概念

## 3.1 消息和批次

消息是kafka的数据但愿，也就是消息中间见的消息，由字节数组组成。通常来说，消息采用序列化和反序列化技术来传输，格式通常为JSON。

## 3.2 主题和分区

Kafka的消息通过主题来分类。主题下可以分为多个分区。有新消息是，会以追加的方式写入分区。因为主题有多个分区，所以在整个主题的范围内无法保证消息的顺序，只能保证单个分区的顺序。

## 3.3 Broker

一个Kafka服务器就是一个Broker。Broker的作用是接收生产者消息、设置偏移量、消息持久化等。

如果存在多个broker，可以构成集群。集群会选举一个Broker作为控制器，用来管理其他broker，如分区分配和监控。

## 3.4 消息示例

比如现在有三台Broker、6个分区，那么Kafka会将分区均匀地分配给Broker。比如B1负责P0、P3，B2负责P1、P4，B3负责P2，P5。然后生产者向任意一个分区发送消息。根据消息内容计算哈希值，得到消息发送到P0。那么消息就会发送给B1。

假如存在两个消费者组，每组两个消费者，那么每个消费组需要平均分配分区，即每个消费者组的每个消费者平均负责3个分区。假如第一组的C1-1负责P0，就会拉取信息写入到数据库。第二组的C2-2负责P0，也会写入到数据库。而偏移量保存在分区中，这样C1-1读取数据后分区记录偏移量，如果分区发生更换，新的消费者就能从原消费者上的偏移位置继续读取。
