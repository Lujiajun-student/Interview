# 1. 40亿个QQ号，限制1G内存，怎么去重

QQ号用int来保存，如果直接存储的话，需要10G内存。因此针对大数据量去重，可以用位图BitMap来实现。用0和1表示某个QQ号是否存在。这样的话，1bit保存一个QQ号，`4*1G*1bit/8=500MB`，内存能够存下。

Redis支持BitMap，于是可以在Redis去重。通过`SETBIT myKey 1384593330 1`就能够用1bit来保存数据。

# 2. 订单超时后自动取消怎么实现

分布式的解决方案是使用Redis和MQ。

Redis可以使用有序集合，键是订单号，值是过期时间戳，循环查看集合中第一个元素是否过期，过期就移除元素再处理。

或者使用Redisson延时队列，RDelayedQueue，下订单的时候同步将订单消息和过期时间发送到延时队列，只要时间一到就会自动发送消息，消费者只需要监听这个延时队列就可以了。

或者使用MQ。如RabbitMQ的死信队列，为消息添加过期时间，只要消息过期，就会进入死信队列，然后让消费者处理死信队列的消息就行。或者RocketMQ，给消息添加延时，过了时间后才发送给消费者。

# 3. 多次输错密码如何限制登录

限制登录需要限制当前账号和ip。可以使用Redis来实现。

在Redis设计关于user的key，键为登录次数。如果登录成功，就把Redis关于用户的这个key给删除掉。如果登录失败，查找Redis是否存在这个用户key，有的话就让登录次数自增，没有的话就创建。如果登录次数达到阈值，就返回账号锁定，把请求给拦截。

还要可能需要限制ip的登录。同样的在Redis中保存当前ip的登录次数，登录成功就清除，登录失败就登录次数自增。达到一定次数也同样锁定账号，拦截请求。

# 4. 秒杀场景下，如何保证最后一件商品只有一个人能抢到

这种情况需要考虑数据库和Redis。

数据库需要使用行级锁和乐观锁，行级锁用来保证性能，乐观锁的话就是在表中添加版本字段，只有修改前后版本号匹配，才能抢，如果不匹配就说明商品被抢走，操作失败。

还有Redis使用分布式锁。可以使用Redisson的tryLock来尝试加锁，获取锁后再扣减库存。或者用Lua脚本，Lua脚本在Redis是原子性的，能够直接用来扣减。

# 5. 如果Session存储在服务器A上，那么访问服务器B的时候，怎么获取Session的信息

这个问题是数据的作用域太小了，保存在一台服务器上导致其他服务器获取不到数据。比较好的方法就是保存在Redis中，服务器读写Session时都保存在Redis中，这样就能保证Session对所有的服务器可见了。或者也可以保存在数据库，但读写性能就低了一点。

# 6. 大文件上传问题

大文件如果上传途中遇到网络问题导致失败，可能需要重新上传。而我们需要使用分片上传的机制做到只需重新上传先前未上传的部分即可。File对象有slice方法，能够把大文件切成固定大小，然后在后端存储已经上传的分片索引，这样根据索引来决定上传哪些分片。全部上传完毕后，用RandomAccessFile来按顺序将分片都写到目标文件。

# 7. 怎么设计秒杀系统

秒杀系统需要满足高并发、高性能和高可用三个方面。

高性能的话，需要处理热点数据。热点数据可以保存在Redis中，或者用Caffeine做一级缓存。

高并发的话，可以将静态资源部署到CDN中，这样大量用户访问静态资源不会经过服务器，减少了服务器的压力。或者在网关Nginx中用漏桶算法来限制访问频率。

高可用，就是要部署集群，像Nginx集群，Redis集群等。还有流量过大时需要做熔断和降级，下游服务过慢就返回系统繁忙的提示，让用户重新尝试。或者做限流也可以，只允许一定数量的请求进入，其他请求需要排队。或者也可以用消息队列来做，将请求放到消息队列，前端直接返回处理中，后端可以根据自己的速度来从消息队列获取消息来处理。

除了这三点，还要保证数据的一致性。减库存通常是下单后就扣减库存，这时候需要避免超卖问题。超卖的话可以使用Redis+Lua脚本来实现，把查询、修改和写回作为一个原子操作，就不会出现超卖现象。然后数据库层面就使用乐观锁，修改数据的时候看修改前后版本号是否一致，一致的话就修改，不一致就返回失败。

还有需要保证幂等性，防止重复下单。在Redis中保存用户和商品id，确保每个用户在一个活动中只能扣除一件商品。

# 8. 排行榜

排行榜要求的是高性能排序和范围查询。

一种排序方法是查询数据库时，用ORDER BY来排序。这种方法的好处是代码简单，不需要引入其他组件，成本低。坏处就是生成排行榜很耗时，消耗数据库的性能。

还有的方法就是使用Redis的ZSet有序集合来处理。有序集合是通过跳表和哈希表来实现的，能够以O(LogN)的时间复杂度来实现插入、更新和查询。

插入的话使用`ZADD orderBoard <score> <user_id>`来实现，通过`ZERVRANGE orderBoard 0 99 WITHSCORES`来获取前100名的排序结果等。

# 9. 站内消息系统

站内消息主要是消息存储和查询的问题。

首先消息主要有三类。第一类是系统消息，属于一对多。第二类是活动消息，属于多对一。第三类是私信，属于一对一。

然后，需要决定消息的存储方式。主要有推和拉两种。推是生产消息时，为将消息发送给每一个接受者的收件箱里。这样的话，每个用户只需要查询自己的收件箱就知道有没有新的消息。但性能不好，生产一条消息时需要修改大量接收者的记录。还有拉模式，生产消息时保存在生产者中，用户登录时自动拉取新的消息。这样发送消息时只需要插入一条记录，但需要维护每个用户读到哪一条信息，而且用户多的时候会有查询压力。

而且这里需要分为在线用户和离线用户，离线用户就用推或者拉的方式存储消息，而在线用户可以通过WebSocket来实时推送。还有未读消息，如果全盘扫描没有读过的消息会浪费时间，只需要在Redis维护计数器，获取新消息时自增，查看消息时清零就行。

从整体上看，产生消息时，消息中心通过MQ来异步处理，根据消息类型选择推或者拉的方式写入数据库，如果用户在线，就发送实时弹窗，并更新未读消息的计数器。

# 10. 如何获取用户的ip归属地

用户的网络请求的request中可以调用`request.getRemoteAddr()`来获取ip，或者通过`request.getHeader("x-forwarded-for")`来获取。

获取到ip地址后，可以使用离线ip数据库或者在线API接口，传入ip就能返回归属地。
