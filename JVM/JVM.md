# 1. JVM内存结构

JVM主要分为栈、堆、方法区、程序计数器、本地方法栈五个部分。

## 1.1 栈

栈是线程私有的，每个方法在执行的时候都会在栈中创建一个栈帧来保存局部变量表、方法出口、操作数栈等信息。

## 1.2 堆

堆是线程共享的，用来保存对象实例，所有的对象都要在堆上分配，以及字符串常量池也会保存在这里（字符串被频繁使用，放到方法区难以回收，只有放到堆中才好通过GC回收）。

## 1.3 方法区

方法区是线程共享的，用来存储常量、静态变量、类信息等。

## 1.4 程序计数器

这是线程私有的，存储线程执行的字节码的行号。

## 1.5 本地方法栈

线程私有，保存使用到的本地方法。

# 2. 对象的创建流程

## 2.1 类加载检查

JVM遇到new指令时，会先检查当前这个类是否被初始化过，初始化过就直接使用，没有的话需要执行类加载过程。

## 2.2 分配内存

类加载检查通过后，会为这个新对象分配内存。

## 2.3 初始化

内存分配完成后，JVM会将分配到的内存空间初始化为0值，保证对象实例的字段不需要赋初始值。

## 2.4 设置对象头

JVM会配置对象，将信息放到对象头。如对象的哈希码、锁状态等。

## 2.5 执行初始化方法

JVM接下来就会执行init方法，根据程序员的意愿来初始化。先执行父类的init方法，然后执行类的赋值语句和代码块，最后就是执行类的构造方法。

# 3. G1垃圾收集器

在G1时期，整个堆内存会划分伊甸区Eden、存活区Survivor、老年区Old、存储大对象的区Humongous。G1使用的是标记整理法。

G1的运行过程有四个。

1. 年轻代GC。Eden区满时就会触发GC。这是STW过程，将Eden和Survivor区存活的对象移动到新的Survivor区或者老年区。
2. 并发标记周期。如果堆内存使用率达到阈值，G1会启动并发标记，准备压缩。首先标记GC root直接引用的对象，然后再扫描Survivor指向老年代的引用，然后再处理并发阶段存在的变动，然后再筛选回收。
3. 混合回收。并发标记后，G1会知道哪些区域的垃圾最多。混合回收时不仅会回收所有的年轻代，还会回收部分老年代。
4. Full GC。如果混合回收不够用，G1会退化为Full GC。

# 4. 类加载机制

类数据会从Class文件加载到内存，并校验数据和初始化来形成JVM可以使用的Java类型。

一个类型的生命周期是加载、验证、准备、解析、初始化、使用、卸载。其中验证、准备和解析称为连接。

# 5. 双亲委派模型

双亲委派模型有四层，从低到高分别是自定义类加载器、应用程序类加载器、拓展类加载器、启动类加载器。如果一个类加载器收到了类加载的请求，它首先会把这个请求委派到父类加载器去完成，只有父类加载器无法完成加载请求时，子加载器才会尝试自己完成加载。

# 6. JVM的整个结构

JVM包含两个子系统和两个组件。子系统分别是类加载、执行引擎，两个组件为运行时数据区、本地接口。

1. 类加载器。负责将编译好的.class字节码文件加载到内存中。会验证字节码的安全性，通过加载、链接和初始化来加载类文件。
2. 执行引擎。作用就是将内存中的字节码翻译为机器码。有即时编译器JIT和垃圾回收器。
3. 运行时数据区就是JVM的内存区域，包含栈、堆、方法区、程序计数器、本地方法栈五个。
4. 本地接口。是与其他语言用来交互的接口，用来调用本地方法。

## 6.1 程序计数器

程序计数器是用来保存当前线程执行的字节码指令的行号。由于JVM多线程时会轮流切换线程，所以为了线程切换后能够回到原本执行的位置，每个线程就都会有一个程序计数器来保存地址。这是线程私有的。

## 6.2 虚拟机栈

虚拟机栈也是线程私有的，生命周期和线程一样。线程执行的每个方法都会在栈里创建栈帧，用来保存这个方法相关的局部变量表、操作数栈、方法出口之类的信息。

## 6.3 堆

堆是JVM内存最大的一块，是线程共享的。JVM规范里的描述是所有的对象实例以及数组都需要在堆上分配。而且堆是垃圾回收主要的区域，称为GC堆。从内存回收的角度看，堆可以分为新生代和老年代。而且JVM的规范中，堆可以处于物理上不连续的内存空间中，只有堆中没有内存可以分配，以及堆无法拓展时，才会抛出OOM错误。

## 6.4 本地方法栈

本地方法栈保存C或C++的代码，用来给程序调用本地方法的。

## 6.5 方法区

方法区是所有线程共享的内存区，保存了类信息、常量、静态变量等数据。

## 6.6 执行引擎

执行引擎负责执行字节码文件，将字节码文件编译为机器码后执行。

# 7. 直接内存

直接内存就是物理内存和虚拟机内存之间的内存。程序读取数据时，通常会从磁盘读取到系统缓冲区，从系统缓冲区再读取到Java缓冲区，然后才能写入到程序中，经过了两次拷贝。而如果使用直接内存，从磁盘文件读取到直接内存，然后Java代码可以直接访问直接内存获取数据，就只需要一次拷贝来获取数据。

# 8. 垃圾回收

Java中，对象的内存是不需要手动释放的，而是由虚拟机自动执行。JVM中存在一个垃圾回收线程，只有虚拟机空闲或者堆内存不足时才会执行，扫描没有引用的对象并添加到需要回收的集合中进行回收。

## 8.1 GC是什么，为什么需要GC

GC就是垃圾回收，GC可以自动检测对象是否没有被引用，没有引用的话就可以被直接回收并释放内存，避免内存越用越少。

## 8.2 垃圾回收原理

在程序员创建对象时，GC就会监控这个对象的地址、大小等信息。GC会用有向图的方式来记录所有对象。如果对象A引用了对象B，那么有向图里的节点A就会指向节点B。这样的话就能从根节点GC Root开始，遍历有向图来获取所有有引用的对象，然后将剩下没有引用的对象回收就可以了。

程序员可以通过System.gc()来手动通知一次垃圾回收，但并不保证垃圾回收会执行。

## 8.3 JVM引用类型

1. 强引用。GC不会回收强引用对象。
2. 软引用。内存溢出才会被回收。
3. 弱引用。GC会直接回收。
4. 虚引用。无法获得对象，GC时会返回一个通知。

## 8.4 怎么判断对象是否可以被回收

有两种方法。第一种是从GC Root出发，遍历有向图搜索所有对象，这些对象都是有引用的对象，然后将剩下没有引用的对象全部回收。

第二种是引用计数法。如果一个对象存在引用，计数器就+1。那么回收的时候，可以根据计数器的值来判断当前对象有没有引用，计数器为0就回收。但这种方法无法解决循环引用的问题。

## 8.5 Full GC

Full GC是对整个堆空间执行一次垃圾回收。由于空间非常大，而且为了避免扫描途中的对象引用发生改变，执行的时候会触发Stop the World，导致所有线程都需要暂停，对性能影响非常大。

堆空间中，永久代不会发生垃圾回收。如果永久代满了，触发的是Full GC/

## 8.6 垃圾回收算法

1. 标记-清除算法。从GC Root开始，遍历并标记所有存活对象，然后统一回收所有没有标记的对象。因为这种方法只会单独清除对象，导致产生大量的内存碎片，如果需要分配大对象，可能还会触发新的GC。
2. 标记-复制算法。内存只使用一半，GC时将存活的对象复制到另一半，然后将这一半的所有对象全部回收。这种方法的内存利用率只有50%。
3. 标记-整理算法。标记所有存活对象，让所有存活对象移动到一端，将剩余的空间全部回收。这种移动会需要开销。
4. 分代算法。将堆内存划分为新生代和老年代，新生代采用复制算法，老年代采用标记整理算法。

## 8.7 新生代、老年代和永久代

Java中，堆划分为新生代和老年代。新生代里面划分了Eden区、From Survivor和to Survivor。

新生代保存的是新创建的对象。首先，对象会放到Eden区。如果Eden满了，就会触发一次Minor GC，扫描Eden和From Survivor，将存活的对象复制到To Survivor，并将存活对象的年龄加1，然后清除掉Eden和From Survivor。这样，Survivor的身份就转变，只有From Survivor用来存储对象。如果有对象的年龄达到15，那么JVM就认为当前对象长期存活，从新生代移动到老年代。或者Minor GC时，遇到大量的存活对象，Survivor放不下，也会移动到老年代。超大对象在存储时直接进入老年代。如果老年代满了，就会触发Full GC，触发STW，对整个堆进行垃圾回收。如果回收完还是不够，就会报错OOM。

## 8.8 Minor GC, Major GC, Full GC是什么

Minor GC是新生代GC，是Eden区满的时候触发的。由于Java对象大多朝生夕死，Minor GC非常频繁，回收速度非常快，采用的是复制算法。

Major GC是老年代GC，通常Major GC连着Minor GC一起执行。

Full GC是清理整个堆空间，包含年轻代和老年代。

## 8.9 为什么新生代分Eden和两个Survivor

因为Java对象朝生夕死，如果Eden区满了后直接送到老年代，老年代很容易被填满，触发Major GC，开销很大。因此需要保证对象尽可能地少进入老年代，进入老年代的对象尽可能地不被回收。因此，需要通过Survivor区来筛选对象，只有对象存活的足够久，才会被送到老年代。而且Survivor使用标记复制算法来移动对象的，能够保证存活对象占用连续的内存空间，避免内存碎片的产生。

# 9. 类加载机制

程序使用某个类是，需要将这个类加载到内存中。主要通过加载、链接和初始化三个步骤。

1. 加载。将类的class文件读取到内存。首先通过类名获取这个类的二进制字节流，通过这个字节流找到类，在方法区中创建运行时数据结构，然后在堆中创建这个类的对象作为访问类的入口。
2. 验证。有了类对象后需要保证这个类是满足要求的，需要验证这个类的文件格式，元数据，字节码等。
3. 准备。类安全后，为类的静态变量在方法区中为静态变量分配内存并设置初始值，如果静态变量是常量，就直接赋值。
4. 解析。将常量池的符号引用替换为直接引用。原本的引用是用一组符号来描述的，如类名，方法名等，在这里需要替换为物理地址。
5. 初始化。执行类的构造器方法，包含静态变量的赋值和静态代码块。

## 9.1 类加载器

1. 启动类加载器。位于最顶层的类加载器。
2. 拓展类加载器。
3. 系统类加载器。
4. 用户自定义类加载器。

这里按照双亲委派原则，首先加载的是最顶层的类，如果找不到才向下寻找。

# 10. 内存泄露

虽然GC很强大， 但依然会存在内存泄露的情况。

如果有的对象的生命周期其实很短，但被长生命周期的对象引用，那么这个短生命周期的对象本应该结束并回收，但由于被长生命周期的对象一直引用，导致标记算法和引用算法都不会清除这个还有引用的对象，这种逻辑上不在使用，物理上可达的情况就是内存泄露。

比如静态引用类型可能本身只短时间使用，但由于是静态，只会在类的结束后才回收，这就导致引用类型对象不会被回收。或者数据库连接、打开文件后没有关闭，也会出现内存泄露。最好的方法就是通过tryWithResource或者finally中保证资源的关闭，也可以监控堆内存，如果内存只增不减，就可以认为出现了内存泄露。
