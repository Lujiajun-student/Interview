# 1. Agent和大模型的区别是什么

普通的大模型本质是一个生成式大模型，给定输入，它会预测接下来的输出。而chat版本的就是给定一个问题，它会给出回答。但这种是被动的，它只会给出输出，不能做事情，而且知识是固定的，不通过训练或者RAG，知识就不会更新。

而Agent包含了更多的功能。Agent = 大模型 + 规划 + 记忆 + 工具。Agent能够将复杂的任务进行拆解，比如我要访问什么网站，我要怎么组织内容，我要使用什么工具之类。

比如使用工具，需要通过提示词告诉大模型工具怎么使用，也就是应该返回什么样的结构数据，这样根据这里的返回数据，能够获取需要执行的方法以及参数，然后执行这个方法，将结果返回给大模型，大模型就获取到需要的结果。这样，大模型实现了工具调用。

而记忆的话，需要设计短期记忆和长期记忆。短期记忆可以存在上下文发送给大模型，长期记忆的化，需要用向量数据库来存储，需要的时候通过语义检索获取。

还有规划。这需要实现多步推理和自我纠错。复杂任务很难一步达成，需要将问题分为多个步骤。假如Agent在执行某一步时出错了，它需要能够检测到失败并分析原因，换一种方式来重试。

# 2. Agent的基本架构

有大脑、规划、记忆和工具使用四个。

大脑就是大模型本身，给定写好的系统提示词，大模型就能分析用户输入，判断下一步应该做什么，是需要规划，调用工具，寻找知识，还是可以结束。而其他三个部件使用后，都需要将结果告知给大模型。

规划就是用一个模型，将复杂任务拆解为多个子任务。比如帮我查找一篇论文，并写出示例代码，就需要有检索资料、分析知识、编写脚本等任务。在这个过程中，agent需要根据自己的历史行为来指出错误并进行优化。这可以使用ReAct或者Reflexion。这个规划过程依赖模型的推理能力，有两种方法能够提升推理能力。

1. CoT思维链。在提示词中要求模型给出最终答案之前，需要先输出中间的推理过程。这样，大模型的输出就会呈现从A到B，再到C，然后输出结果。思维链能够提升大模型处理复杂任务的准确率。但如果其中一步推导错了，接下来就会越来越离谱。
2. ToT思维树。它认为复杂问题的单一推理是不够的，需要产生多个推理方案，寻找最优的方法来继续推理。步骤为：在当前状态下生成多个可能的下一步，评价每个下一步对最终目标的价值，然后寻找最优路径，根据最优路径来进行推理。

现在的规划主要有两种模式，一是先规划后执行，让LLM输出完整的步骤，然后按顺序执行。二是边执行边规划，每走一步就思考下一步应该干什么。

记忆分为长期记忆和短期记忆，短期记忆能够放到上下文，长期记忆就放到向量数据库，有需求的时候就检索。

短期记忆最大的问题是上下文有限，如果对话过多，需要把前面的上下文做摘要浓缩，或者做滑动窗口。

长期记忆最主要是选择什么应该存，什么不应该存，以及存了之后怎么管理。最好的方法是在存储的时候做重要性评估，只持久化重要的信息。

而工具是用来告诉模型工具叫什么名字，能做什么事，需要什么参数。交给agent后，agent决定调用时，就会从工具中选择工具，把决策通过json的格式给出来，交给其他函数执行，结果再反馈给模型。

这样的话，agent就会运行成下面的伪代码过程。

```python
def agent_run(user_goal: str):
    # 规划模块，将目标拆成步骤列表
    plan = llm.plan(user_goal)
    memory = [] # 短期记忆
    for step in plan:
        # 做决策
        action = llm.decide(
        step=step,
        history=memory,
        long_term=vector_db.search(step)
        )
        if action.type == "tool_call":
            result = tools.execute(action.tool_name, action.args)
            memory.append({"step": step, "result": result}) # 执行结果存入记忆
        elif action.type == "final_answer":
            # 执行结束，最终返回
            return action.content
```

# 3. Tool

Tools是一个工具，需要给大模型看，大模型就能知道它能够调用哪些工具。这个tools给出能够调用的方法、描述、输入和输出规范。

```python
tools = [
    {
        "name": "web_search",
        "description": "在互联网上搜索信息，适合查询实时数据",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "搜索关键字"}
            },
            "required": ["query"]
        }
    },
    {
        "name": "send_email",
        "description": "向指定邮箱发送邮件",
        "parameters": {
            "type": "object",
            "properties": {
                "to": {"type": "string", "description": "收件人邮箱地址"},
                "subject": {"type": "string", "description": "邮箱主题"},
                "body": {"type": "string", "description": "邮箱正文"}
            },
            "required": ["to", "subject", "body"]
        }
    }
]
```

上面就是定义了工具，里面写了大模型能够调用的方法。如果模型返回的是send_email，那么后端截获了请求后，就要调用这个方法，根据properties里的内容来发送邮件。

# 4. Agent

Agent就是有了工具，需要决定使用什么工具的角色。

比如给Agent一个目标，帮我查询最近的天气，那么agent根据工具，会返回一个关于web_search的json文件，然后后端获取这个请求，执行了搜索方法，将搜索结果添加到当前状态，agent再做下一步决策。这里agent和工具方法可以构建成有向图，agent指向所有工具方法，所有方法又指向agent，这样agent就能根据需求来调用所有工具。

但调用不能无限调用，除了agent认为任务完成，给出结束工作流的情况，还需要加多停止机制。

1. 最大循环次数。
2. 计算token累积。
3. 限制执行时间。

还有，Agent有另一个副作用，行为是随机性的。每次调用的推理过程都不一样，最终结果可能也会不一样。在调用方法后，使用日志记录agent的思考过程和工具调用结果。
